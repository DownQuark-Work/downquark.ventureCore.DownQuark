about From the docs:
What should Utilities do?

  3. Abstract repeatability away from the CSS and apply it in the HTML instead

Block Styles

about From the docs:
What should a Block do?
  1. Extend the work already done by the global CSS, composition and utility layers
  2. Apply a collection of design tokens within a concise group
    3. Create a namespace or specificity boost to control a specific context
What shouldn’t a Block do?
  1. Grow to anything larger than a handful of CSS rules (max 80-100 lines)
  2. Solve more than one contextual problem. For example: styling a card and a button in one file
Think of the composition as a skeleton.

TLDR: A block is a skeletal component or organisational structure.
Making it too linear for the dq namespace, but including the description for completeness.

Exception Styles

about From the docs:
Exceptions are often caused by outside influences (such as JavaScript), and using only data attribute selectors (that both CSS and JavaScript can use efficiently) makes sense.
For example: you might have a “reversed” state or an “inactive” state. We use data attributes because an exception should only occur in exceptional circumstances (the clue is in the name).
...
What should an exception do?
  1. Provide a concise variation to a block
  2. Use data attributes
What shouldn’t an exception do?
  1. Variate a block to the point where it isn’t recognisable anymore.
    • This is where a new block should be created
  2. Use CSS classes
Usually, an exception is related to a state change

TLDR: An exception is a deviation from the rules outlined in a block.
Making it too linear for the dq namespace, but including the description for completeness.
